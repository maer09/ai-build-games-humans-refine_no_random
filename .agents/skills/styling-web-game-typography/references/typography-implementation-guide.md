# 排版实现指南（引擎无关）

分布式游戏（Web 导出、可下载、打包）的排版实现指南。涵盖基于主题的角色分离、分阶段字体采用和许可证操作，同时与整体视觉设计保持一致。

引擎专项实现细节（如 Godot `Theme` / `draw_string()`）见同一技能下的对应参考文档（Godot 4.2+ 见 `references/godot-patterns.md`）。

## 1. 目的

- 使用与游戏概念相同的视觉语言设计 UI/HUD/效果文本。
- 平衡可读性与符号风格和氛围。
- 防止再分发（Web、可下载包）中的许可证问题。

## 2. 设计原则

1. 角色分离
   - 将显示角色拆分为 `Heading / Info / Numeric / Emphasis`。
   - 为每个角色固定尺寸/字重/颜色/运动规则。

2. 一致性
   - 通过匹配笔画质感、色温和运动质感，让文本与整体视觉处于同一世界。

3. 可读性优先
   - 对于关键信息（HP、得分、警告），优先保证可读速度而非装饰。
   - 在嘈杂背景上，通过描边/阴影确保对比度。

4. 状态传达
   - 将文本用作状态指示器，而非装饰。
   - 为得分获得、伤害、危险、连击等提供视觉反应。

## 3. 分阶段实现流程

排版通常分两个阶段实现：早期实现阶段的回退字体阶段，以及游戏手感和视觉方向稳定后的完整字体采用阶段。

更新现有项目时，先检查其当前的主题/样式目录、排版文档、字体目录、许可证目录和 `THIRD_PARTY_LICENSES.md`。如果已有约定则保留。如果指定的项目路径不存在，不要编造项目盘点；生成一个基于标准路径的计划并明确说明无法检查本地项目。

### 初始实现

- 仅使用引擎的默认/回退字体实现（不打包外部字体）。
- 在主题/样式资源中定义基于角色的尺寸/颜色 token（见 §4）。
- 字体槽位保持空置，先完成布局/效果逻辑。
- 在 `TYPOGRAPHY_DECISION.md` 中记录字体尚未选定，当前使用回退字体。
- 不要为仅使用回退字体的排版创建 `licenses/fonts/` 或在 `THIRD_PARTY_LICENSES.md` 中添加字体条目。
- 如果 `VISUAL_DESIGN.md` 已存在，其中仅保留简短的排版策略：角色、尺寸层级、颜色和事件文本效果。

### 完整字体采用

- 使用人类反馈对比和选择字体。
- 选择各角色所需的最少字重，然后仅将这些已采用文件打包到 `assets/fonts/`（不要批量打包候选字体）。
- 如果字体来自临时/下载文件夹，复制前须验证确切的来源版本、文件身份和许可证。如果来源无法确认，需从权威来源重新获取仅选定的字重。
- 按照 §7 完成许可证检查和打包。
- 在 `TYPOGRAPHY_DECISION.md` 中记录选择理由/对比/许可证信息。
- 更新 `THIRD_PARTY_LICENSES.md` 和 `licenses/`。

## 4. 通过主题/样式系统统一管理字体

将引擎的主题/样式系统视为项目的排版单元。以下 token 名称是建议；将其映射到引擎对主题/样式原语的实际命名上。

### 4.1 推荐 Token 集

- 字体族：
  - `font_ui_base`（正文/通用）
  - `font_ui_display`（标题/效果）
  - `font_ui_numeric`（得分/计时器）
- 尺寸：
  - `size_xs`、`size_sm`、`size_md`、`size_lg`、`size_xl`
- 颜色：
  - `text_primary`、`text_muted`、`text_positive`、`text_warning`、`text_danger`
- 效果：
  - `outline_size`、`outline_color`、`shadow_size`、`shadow_color`

### 4.2 操作规则

1. 先在主题/样式资源中全局定义；尽量减少逐元素覆盖。
2. 标题/正文最多使用两个字体族；加上数值字体最多三个。
3. 相同语义含义使用相同 token（如所有警告文本使用 `text_warning`）。
4. 避免破坏可读性的装饰（过度发光、重度模糊）。

## 5. 实现模式

### 5.1 基础

- 在主题/样式资源中设置字体并全局应用到 UI 元素。
- 仅在例外情况下使用逐元素字体覆盖。
- 数值 HUD 使用等宽或固定宽度字体。

### 5.2 实用样式

- 边缘强调：小描边用于与背景分离。
- 层次感：微妙阴影增加深度。
- 状态颜色：仅在危险时偏移到暖色调。
- 动态显示：仅在连击时短暂改变尺寸/位置。
- 上下文装饰：仅在重要事件时强调。

### 5.3 选择程序化效果文本

大多数引擎区分两种文本渲染路径：

- **基于节点/控件**（高级 UI 原语，如 Label）：可维护性优先。
- **程序化/即时模式绘制**：效果优先，用于一次性动画文本，如得分弹出、`GAME OVER`、连击提示。

规则：信息性文本使用节点/控件方式；效果文本使用绘制方式。

### 5.4 程序化绘制布局注意事项（重要）

通过引擎的程序化/即时模式绘制 API 绘制文本时，以下陷阱在大多数引擎中普遍存在：

- 绘制 `position` 通常基于基线，而非视觉中心。
- 引擎的"居中对齐"标志在 `width = -1`（或等效值）时往往不会如预期居中。
- 对于居中固定文本（如 `GAME OVER`），使用以下方式之一：
  1. 设置 `width = viewport_width` 并居中对齐绘制。
  2. 测量文本并手动应用 `x -= width / 2`。
- 使用字体的 ascent/descent 进行正确的垂直居中，而不是将基线当作中心。
- 额外的阴影/描边绘制会将视觉中心偏移到右下角；需补偿合成偏移，或使用对称描边绘制，然后在屏幕上验证最终位置。

引擎专项的最小居中模式（Godot 4.2+ `draw_string()`）见 `references/godot-patterns.md`。

## 6. 映射到视觉标签

- `typography-*`：字体选择、间距和运动行为的主要维度
- `render-*`：描边/填充/线条质感的一致性
- `lighting-*`：与发光/叠加表现形式的兼容性
- `analog-*`：仅在事件时使用微妙的抖动/色差
- `composition-*`：文本放置的留白和视线引导

## 7. 再分发的许可证操作

### 7.1 推荐许可证

- `SIL Open Font License 1.1 (OFL)`
- `Apache License 2.0`
- `Ubuntu Font Licence`

### 7.2 采用前检查清单

- [ ] 再分发（打包）被明确允许
- [ ] 嵌入使用被允许
- [ ] 商业使用条件已满足
- [ ] 修改后的命名条件已了解（如 OFL RFN）
- [ ] 许可证文本打包要求可以满足
- [ ] 确切的来源 URL/版本和打包文件名/字重已记录
- [ ] 临时/下载文件已与来源版本匹配，或已替换为从权威来源获取的文件

### 7.3 打包步骤

1. 仅将选定字重放入 `assets/fonts/`。
2. 将对应许可证文本放入 `licenses/fonts/`。
3. 验证导出设置，确保非资源文本文件不会被丢弃（此项因引擎而异；Godot 见 `godot-patterns.md` 的 Web 导出过滤章节）。
4. 在 `THIRD_PARTY_LICENSES.md` 中记录字体名称、来源 URL/版本、打包文件名、字重和许可证。
5. 确认构建不再仅依赖引擎的回退字体；已显式加载已采用的字体文件。

## 8. 推荐目录结构

```text
<PROJECT_DIR>/
  assets/fonts/
    UiBase-Regular.ttf
    UiDisplay-Bold.ttf
    UiNumeric-Semibold.ttf
  themes/                 # 或 styles/，取决于引擎约定
    default_theme.<ext>
  licenses/fonts/
    OFL-UiBase.txt
    LICENSE-UiDisplay.txt
```

## 9. 每个游戏的必需交付物

对于每个游戏项目，在项目目录下记录排版决策和许可证状态。

- `TYPOGRAPHY_DECISION.md`
  - 记录回退/采用状态、角色映射、选定字体文件、选择理由和许可证状态。
- `VISUAL_DESIGN.md`
  - 仅记录简短的排版策略摘要（角色、尺寸层级、颜色、效果规则）。
- `THIRD_PARTY_LICENSES.md`
  - 记录已采用的外部字体来源和许可证；仅使用回退字体时省略字体条目。
- `licenses/`
  - 仅包含已采用外部字体的完整许可证文本。

## 10. 审查清单

§2 设计原则（角色分离、一致性、可读性优先、状态传达）涵盖了基础检查。在此基础上，还需验证：

- [ ] 分发目标的许可证打包已完成
- [ ] 居中固定文本（如游戏结束）在实际渲染中视觉居中（程序化绘制的对齐/宽度已检查）
- [ ] 基线推导的 Y 偏移在可接受范围内
- [ ] 非视觉测试（headless / 单元测试）通过后，UI 布局在实际导出目标中仍然正确

## 11. 最终目标视觉检查（必需）

Headless 或非视觉测试可能遗漏排版定位偏差。导出到真实目标（Web 构建、打包二进制文件）后，至少检查以下内容：

- HUD 在 16:9 和较窄窗口中都保持可读。
- 居中文本（如 `GAME OVER`）确实居中。
- 装饰（阴影/描边/颜色分离）不影响可读性。
- 最终渲染的合成结果是居中的，不仅仅是填充字形；如有需要，使用临时居中辅助线或截图叠加来验证。

## 12. 反模式

- 未经许可证验证就打包字体
- 假设引擎的居中对齐标志就能保证居中，跳过视觉确认

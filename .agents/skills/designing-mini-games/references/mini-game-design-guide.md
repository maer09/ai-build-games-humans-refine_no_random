# 迷你游戏设计指南

## 1. 设计挑战

- 以最少的输入创造多样化的游戏体验。
- 设计合适的难度曲线和风险/收益系统。
- 为玩家操作提供直观的反馈。
- 防止单调操作（连按、挂机）。

## 2. 四项核心设计原则与评估标准

将"原则（该做什么）"和"评估（确认事项）"整合，明确设计方针。

### (1) 简洁与直观

- 原则：使用基本形状（圆形、三角形、方形），保持背景简洁。消除 UI、说明文字和多资源管理。创造"不言自明"的结构，让规则通过游玩传达。
- 评估：能否在无需文字的情况下立即理解规则和对象的角色？

### (2) 视觉反馈与游戏结束

- 原则：通过动画、颜色和大小变化传达成功、失败和危险状态。游戏结束条件应单一且一目了然，例如"碰撞"或"坠落"。
- 评估：操作结果是否在视觉上清晰可见？失败原因是否公平且显而易见？

### (3) 技巧型得分与风险/收益

- 原则：奖励有意操作和高风险行为（如险象环生），而非简单任务。让熟练度直接反映在得分中。
- 评估：得分是否反映玩家技巧？是否始终存在有意义的选择（安全与挑战之间）？

### (4) 新颖机制

- 原则：不受现有概念束缚，从物理定律（重力、磁力、惯性）、几何原理或其否定出发，发明令人惊喜的行为。
- 评估：
  - 是否有让玩家感到"前所未见"的时刻？
  - 是否存在无法仅用现有标签组合解释的元素？
  - 单一机制是否衍生出多样化的玩法发展？

## 3. 交互模式（参考）

基于输入的机制示例。这些是创意的起点，而非约束。如果项目简报中提供了按键数量限制，请在该范围内设计。

### 3.1 单按键状态模式

| 输入 | 机制 | 应用示例 |
| :--- | :--- | :--- |
| **按下** | 瞬间变化 | 方向切换（90°/180°）、跳跃、射击、传送、分裂、属性切换 |
| **长按** | 积累/延伸 | 力量/角度调节、拉伸、护盾展开、能量蓄力 |
| **释放** | 释放/反弹 | 抛射物发射、蓄力攻击执行、状态释放效果 |

### 3.2 多按键模式（button_types ≥ 2）

| 模式 | 机制 | 应用示例 |
| :--- | :--- | :--- |
| **角色分离** | 每个按键控制不同轴 | 移动/攻击、左/右、跳跃/射击 |
| **互斥切换** | 同时只有一个按键的效果生效 | 姿态切换（进攻 ↔ 防御）、元素循环 |
| **同时组合** | 同时按下多个按键触发特殊动作 | 蓄力冲刺（移动 + 攻击）、紧急制动 |
| **顺序链** | 按键顺序有意义 | 必杀技输入连段、节奏序列 |

## 4. 移动与环境机制（参考）

移动模式与地形组合的示例。欢迎超出这些范围的创意。

### 4.1 玩家移动/操作

- **自动移动**：自动奔跑、持续弹跳、固定振荡、加速
- **特殊移动**：重力反转、墙壁反射、定点旋转、传送
- **操作**：范围攻击（AoE）、反击、基于物理的抛射物、连锁反应、状态切换

### 4.2 环境/地形交互

- **地形**：不规则地面、浮动/移动平台、深渊、临时落脚点
- **机关**：行为会变化的环境区域、危险物（尖刺、碾压机）、物理谜题

## 5. 标签的作用

标签不是设计规格，而是灵感起点（种子）。

- **刺激而非约束**：从标签展开联想，可以自由偏离标签
- **矛盾即机遇**：利用矛盾的标签创造创意张力（参见 §8）
- **允许偏离**：最终设计无法用标签解释也没问题
- **目的**：作为随机性来"偏移" LLM 的想法，使其脱离现有模式

## 6. 创意起点

独立于标签的创意出发点。当陷入僵局或从一开始就追求新颖性时使用。

### 6.1 抽象问题

| 视角 | 示例问题 |
| :--- | :--- |
| **否定** | 如果没有屏幕会怎样？没有分数？没有失败？ |
| **感官** | 什么时刻会让人心跳加速？什么是如释重负？什么是"险象环生"？ |
| **超越物理** | 如果能操纵概率会怎样？如果因果反转？如果时间分叉？ |
| **跨学科** | 音乐的紧张与解决、生态系统中的捕食、化学反应链 |
| **从情感逆向** | 如何创造"背叛"的感觉？什么是"发现"的喜悦？ |

### 6.2 从约束中产生创意

设定"没有~"的约束条件，然后逆向思考：

- 没有移动（仅靠画面变化运作的游戏）
- 没有敌人（与环境或自身对抗）
- 没有得分（目标是维持或改变状态）
- 没有画面（仅靠声音或节奏运作）

## 7. 从标签设计游戏的流程

按照以下顺序，根据给定的标签组进行设计。

1. **自由联想**：查看标签，将脑海中浮现的第一印象或感受用语言表达出来（参考 §6）
2. **偏离探索**：思考标签的"反面"、"否定"或"极端"，探索意想不到的方向
3. **核心体验确定**：用一句话定义你想给玩家的"瞬间感受"
4. **机制构建**：设计实现核心体验的输入方案
5. **一致性验证**：用 §10 的检查清单验证设计

※ 第 1-2 步使用标签作为刺激，第 3 步起不受标签束缚。

## 8. 标签矛盾与创意张力

当遇到矛盾的标签时，这是发明的机会，而非约束。

| 矛盾示例 | 常规处理 | 创意解读 |
| :--- | :--- | :--- |
| `field:1D` 和 `field:3D` | 采纳其中一个 | 看似 1D 但有深度的空间，或在 3D 空间中做 1D 式移动 |
| `on_pressed:jump` 和 `on_pressed:shoot` | 按优先级选择 | 跳跃和射击作为同一动作（跳跃轨迹成为攻击等） |
| `player:auto_move` 和 `on_holding:stop` | 整理依赖关系 | 停止本身就是一种风险的设计 |

**原则**：不要解决矛盾，而是发明一个让矛盾得以成立的新概念。

## 9. 输出格式

按照以下格式输出到项目的游戏设计文档。

```markdown

## 0. 标签记录

- 机制 (3): #m_tag1, #m_tag2, #m_tag3
- 视觉 (2): #v_tag1, #v_tag2
- 结构 (1): #s_tag1
- button_types: <1-5>
- 非显而易见对检查: `<pair_a> + <pair_b>` 满足项目的新颖性或非显而易见性规则

## 0.5 状态模型（最小化）

| 状态变量 | 增减触发条件 | UI/反馈呈现 |
| :--- | :--- | :--- |
| var_a | <什么会改变它> | <在哪里/如何显示> |
| var_b | <什么会改变它> | <在哪里/如何显示> |

注意事项：
- 仅当状态变量能创造现有规则无法表达的新决策时，才添加该变量。
- 每个状态变量必须至少有一种游戏内的呈现方式（地形/行为/颜色/形状/速度/声音），而不仅仅是 HUD 数字。

## 1. 核心机制

<输入 → 行为 → 结束条件，得分系统，难度递增机制>
<难度变量约定：定义初始值、增长节奏和预期范围>

## 1.5 权衡定义

- 具体行为对: `<action_safe>` 与 `<action_risky>`
- 权衡说明: 改善一个状态如何使另一个状态恶化

## 2. 对象规格

<每个对象的形状、行为、碰撞处理>

## 3. 设计指南分析

<对四项核心设计原则的评估>

## 4. 与标签的关系

<从标签出发的创意发展过程>

## 5. 新颖性依据

<超越现有模式的元素，以及本游戏与最相似现有游戏的区别>
```

## 10. 设计质量检查清单

完成设计前确认以下事项。

- [ ] 输入方案是否在项目简报中选择的按键数量限制内？
- [ ] 游戏结束条件是否单一且在视觉上一目了然？
- [ ] 连按/挂机是否不是最优解？
- [ ] 能否对 §2 的 4 项原则都给出有理有据的回答？
- [ ] 创意是否从标签出发，并包含超越现有模式的元素？
- [ ] 是否有让人感到"前所未见"的时刻？
- [ ] 如果使用了状态变量，每个变量是否都有明确的决策目的作为支撑？
- [ ] 如果使用了状态变量，每个变量是否都有非文字的游戏内反馈渠道？
- [ ] 状态/操作之间是否至少有 1 个明确的权衡（包括操作与地形的权衡）？
- [ ] 玩家操作是否至少在游戏世界侧留下了 1 个持久的历史痕迹？

## 附录: SCAMPER 方法（辅助技法）

通过对现有元素的变换来辅助创意构思。不过，这是陷入僵局时的辅助手段，不是主要的创意生成方法。

- **替代 (Substitute)**：用传送或重力反转替代跳跃。
- **组合 (Combine)**：将弹跳机制与方向切换结合。
- **调整 (Adapt)**：将现有街机游戏或物理现象（钟摆、波浪）适配到选定的 `button_types` 约束中。
- **修改 (Modify)**：角色随长按时长变大。危险随速度增加。
- **其他用途 (Put to other uses)**：将敌人用作平台或工具。
- **消除 (Eliminate)**：移除"理所当然"的元素，如重力或直接移动控制。
- **重排 (Rearrange)**：关卡构成不断变化。

**注意**：SCAMPER 方法是对现有元素的变换，在生成根本性新创意方面有其局限。请与 §6 的抽象问题配合使用，以追求超越变换的"新概念"。

# 音效设计指南

在 Godot 4.2+ 中使用程序化音频生成来设计动作迷你游戏的音效层的指南。音效设计的方向从视觉标签推导而来，不使用单独的音效标签。不需要外部音频资源；所有声音均通过 GDScript 在运行时合成。

## 1. 设计挑战

- 在没有预录制资源的情况下生成独特且令人满意的音效。
- 保持音视频一致性，声音必须让人感觉与视觉属于同一个世界。
- 提供清晰的听觉反馈来强化（而非重复）视觉反馈。
- 在迷你游戏范围内保持程序化音频代码紧凑且高性能。

## 2. 五项核心音频原则与评估标准

### (1) 事件清晰度

- 原则：每个重要的游戏事件都必须产生独特的声音。玩家应该能够仅凭声音判断发生了什么，得分、伤害、险些擦过和游戏结束在听觉上必须可区分。
- 评估：玩家能否闭眼识别事件类型？不同事件幅度（1 分 vs 10 分）在听觉上是否有差异？

### (2) 音视频一致性

- 原则：声音风格必须与视觉标识匹配。霓虹发光风格的游戏应该听起来具有合成感和共鸣感；几何原始体风格的游戏应该听起来干净且有打击感。视觉标签同时驱动两个领域。
- 评估：声音是否感觉来自与视觉相同的世界？如果换成通用的 8-bit 声音是否会格格不入？

### (3) 克制

- 原则：少数精心设计的声音胜过许多通用的声音。一个迷你游戏最多需要 4-8 种不同的声音事件。每个声音应该简短（50-500ms）且目的明确。静默是有效的设计元素。
- 评估：是否有任何声音感觉多余？在激烈的游戏过程中是否存在声音杂乱？

### (4) 动态响应

- 原则：声音应该对游戏状态做出反应，连击倍增时音高上升，冲击声音随速度缩放，危险声音随难度增加而增强。静态不变的声音会让人感到死板。
- 评估：声音是否随游戏强度变化？玩家能否仅通过音频感知局势升级？

### (5) 语义锁定（游戏内）与变化（跨游戏）

- 原则：在单个游戏内，每个事件族（`得分 / 危险 / 伤害 / 状态变化`）应保持一致的音色特征以便快速识别。在不同游戏之间，该特征必须从视觉标签和游戏概念重新设计，而非作为全局模板复制。
- 说明：
  - **游戏内固定**：事件含义与音色的映射在一次游戏会话中保持稳定。
  - **跨游戏可变**：波形集、音高范围、包络形状、调制风格和节奏间隔应因游戏而异。
- 评估：在这个游戏中事件含义是否可即时识别？这个游戏的声音与另一个生成的游戏是否有区别？

## 4. 声音事件目录

每个迷你游戏都应为以下事件定义声音。并非所有事件都适用于每个游戏，选择与机制相关的部分。

### 4.1 核心事件（必需）

| 事件              | 用途                   | 典型时长   | 设计说明                                                     |
| :---------------- | :--------------------- | :--------- | :----------------------------------------------------------- |
| **得分**          | 正向强化               | 50-150ms   | 明亮，音高上升。音高/层数随分值缩放。                        |
| **伤害 / 死亡**   | 失败信号               | 200-500ms  | 低沉、不和谐或有噪声。必须与得分声音形成鲜明对比。           |
| **游戏结束**      | 终止状态               | 300-800ms  | 音高下降或淡出。终结感。                                     |

### 4.2 增强事件（推荐）

| 事件                     | 用途                 | 典型时长   | 设计说明                                                     |
| :------------------------ | :------------------- | :--------- | :----------------------------------------------------------- |
| **险些擦过**             | 紧张 / 奖励         | 50-100ms   | 微妙、快速。音高略低于得分声音。                             |
| **碰撞（非致命）**       | 物理反馈             | 30-100ms   | 短促、打击感。音高/音量随冲击速度缩放。                     |
| **状态变化**             | 模式切换             | 100-300ms  | 与其他声音明显不同的音色。                                   |
| **连击 / 倍增**          | 事态升级             | 50-150ms   | 与得分同族，但每步音高递增。                                 |
| **蓄力完成**             | 行动就绪             | 100-200ms  | 积蓄后的明亮 ping。                                          |

### 4.3 环境事件（可选）

| 事件                | 用途                 | 典型时长            | 设计说明                                                     |
| :------------------ | :------------------- | :------------------ | :----------------------------------------------------------- |
| **背景嗡鸣**       | 氛围                 | 持续 / 循环         | 非常低音量。匹配背景视觉风格。                               |
| **难度切换**       | 节奏信号             | 200-400ms           | 微妙的音调变化。玩家可能不会有意识地注意到。                 |

## 5. Godot 4.2+ 中的程序化音频，实现指南

所有声音均通过 GDScript 程序化生成。不使用预录制的 `.wav`、`.ogg` 或 `.mp3` 资源。

本节侧重设计决策。模板资源是实现代码的权威来源。

- 波形基元和简单包络：`.agents/skills/creating-godot-procedural-audio/assets/audio_synth.gd`
- 本仓库中测试驱动的实现示例使用 `.<slug>/main.gd` 和 `.<slug>/tools/tests/run_tests.gd`；根据项目调整路径。

### 5.1 运行时架构（推荐）

默认使用混合方案。

1. **单次触发 SFX（得分 / 危险 / 伤害 / 状态变化）**
   默认使用 `程序化生成 -> PCM 缓存 -> AudioStreamWAV`。
2. **持续可控音频（引擎 / 推进 / 光束等）**
   先尝试 `程序化生成 -> PCM 缓存 -> AudioStreamWAV(循环)`。如果需求可以通过 `volume_db / pitch_scale` 满足，则作为默认方案。
   仅在实时合成不可或缺时才采用 `AudioStreamGenerator` + `AudioStreamGeneratorPlayback`。
3. 使用 `11025` 或 `22050` 作为基准 `mix_rate`（优先考虑 GDScript 负载）。
4. 对同一事件保持固定主旋律并加入微小变化，避免纯随机。
5. **优先使用可采样的路径**：优先使用可被 `AudioStreamPlayer` 作为采样播放的流（如 `AudioStreamWAV`），避免常规使用非采样流（主要是 `AudioStreamGenerator`）。

实现注意事项：

- 将 PCM 缓存保留在 `PackedByteArray`（16 位 PCM）中，转换为 `AudioStreamWAV` 并复用。
- 仅对持续音效监控 `frames_available` / `get_skips()`，必要时重试队列或调整负载。
- 在 headless 运行中（`OS.has_feature("headless")`），跳过 `AudioStreamPlayer` 的创建/播放/缓冲填充，返回空操作句柄，使测试保持纯逻辑。

### 5.2 事件语义锁定（游戏内）

在单个游戏内锁定 `得分 / 危险 / 伤害 / 状态变化` 的音色映射。

- `得分`：明亮的短音，带有上升或开放的印象
- `危险`：中高频噪声或不和谐成分，传达紧张感
- `伤害`：偏向低频 + 短暂噪声瞬态
- `状态变化`：独立主旋律，指示语义转换（通常 120-350ms）

### 5.3 变化预算（推荐范围）

仅在保持可识别性的范围内应用随机化。

- 音高失谐：`±2%-±6%`
- 时长变化：基准的 `0-20%`
- 包络变化：
  - 起音比率：`±0.03`
  - 释放比率：`±0.20`
- 占空比变化（方波/脉冲）：`±0.03-0.12`
- 噪声混合变化：`±0.05-0.15`

### 5.4 持续音效控制（引擎/推进/光束）

分三个阶段控制持续音效。

1. 目标：从输入、速度和危险等级推导目标增益/目标音高
2. 平滑：使用 `current += (target - current) * alpha` 进行平滑
3. 释放：输入释放后，应用约 100-250ms 的衰减

操作规则：

- 如果 `AudioStreamWAV(循环)` + `volume_db / pitch_scale` 能满足需求，作为首选
- 明确停止条件和停止方式（仅增益衰减或带 `clear_buffer()`）
- 对持续音效使用稳定的控制 API：`start_<sound>()`、`update_<sound>(params)`、`release_<sound>()` 和 `stop_<sound>()`。在 headless 模式下，这些调用应更新内部状态并返回空操作句柄，不创建播放器或填充缓冲。
- 避免持续音效与短促音效之间的频段冲突（优先保证得分的可听性）
- 采用 `AudioStreamGenerator` 时，需在 `SOUND_DESIGN.md` 中明确说明为何 WAV 循环无法替代

### 5.5 调试检查清单

- 无声音：
  - 单次触发 SFX：检查 PCM 缓存键解析和 `AudioStreamWAV.data` 长度
  - 持续音效：验证 `play()` -> `get_stream_playback()` -> `frames_available` 的顺序
  - 验证持续音效的事件队列重试是否正常工作
- 削波/失真：
  - 将采样保持在 `[-1.0, 1.0]` 范围内
  - 降低噪声混合比和同时发声数
- 掉音：
  - 监控 `get_skips()` 并调整 `mix_rate` / 同时发声数 / 计算负载

### 5.6 性能模式

1. **高频单次触发应默认缓存**
   - 目标：高频、短促、参数变化小的单次触发 SFX
   - 建议：在启动/加载时生成 `程序化 -> PCM 缓存 -> AudioStreamWAV`，播放时从池中选择
   - 例外：仅当波形本身需要因即时游戏状态显著变化时才允许运行时合成

2. **将局部冷却与全局发射预算分离**
   - 仅靠每事件冷却在事件类型增多时可能导致同时发声数激增
   - 在每类型控制之外，结合 `全局冷却` 和 `每帧发射上限` 来平滑峰值负载

3. **在热路径中使用表驱动配置**
   - 将波形/包络/优先级配置存储在 `const` 表中，避免每次调用重新生成
   - 抑制热路径中基于 `match` 的字典生成和过多的临时对象

4. **持续噪声成分需要状态门控**
   - 不要让持续音效（引擎/推进/光束等）的噪声成分保持恒定；通过速度、危险等级和输入状态进行门控
   - 在低负载状态下减少恒定噪声感，保持信息性声音（得分/伤害）的可听性

## 6. 从视觉标签进行音效设计的流程

在视觉方向完成后进行设计。
如果视觉方向使用自定义或概念衍生的标签，而非 §3 中的精确条目，则将它们映射到最接近的材料、几何、运动或氛围族，并在选择波形之前记录该映射。
示例：`render-luminous-glass` 可以映射到 `render-glow-outline` 加 `geometry-regular-polygon-language`（共鸣正弦/三角波泛音）；`motionviz-reflection-bursts` 可以映射到 `motionviz-impact-ripple` 加 `motionviz-afterimage-trail`（短噪声瞬态进入回响尾音或回声拍）。

1. **标签解读**：阅读每个视觉标签的 `描述` 和 `关键词`。识别它们所暗示的声音特质（参考 §3 映射表）。
2. **跨标签音频综合**：找出所有选定视觉标签之间的共同声音特征。用一句话表达（例如"共鸣的合成脉冲"、"粗糙的打击咔嗒声"）。
3. **波形选择**：选择 1-2 种主要波形和匹配该方向的调制技术。
4. **语义音色映射**：为 `得分 / 危险 / 伤害 / 状态变化` 定义稳定的游戏内映射（相同含义，相同音色族）。
5. **跨游戏变化计划**：明确决定与之前游戏相比会有哪些变化：
   - 波形集
   - 音高范围
   - 包络配置
   - 调制方法（FM/AM/噪声混合/滤波扫频）
   - 节奏/间隔模式
6. **事件映射**：为每个游戏事件（§4），使用选定的波形族和包络风格设计具体的声音。
7. **动态参数**：确定哪些声音参数应对游戏状态做出响应（连击、速度、难度）。
8. **持续音效策略**（如使用）：定义触发条件、停止条件和可接受的释放尾音。
9. **整合检查**：使用 §8 的检查清单验证音视频一致性和事件覆盖率。

## 8. 音效设计质量检查清单

§2 的评估涵盖事件清晰度、音视频一致性、克制、动态响应和语义锁定与跨游戏变化。完成这些评估后，检查以下额外设计执行项：

- [ ] 变化保持在定义的预算内（音高 / 时长 / 包络 / 占空比 / 噪声），使事件特征保持清晰（参见 §5.3）。
- [ ] 使用主旋律优先的变化方式（2-3 个固定变体 + 微变化），而非纯随机。
- [ ] 持续音效在输入/状态结束后在预期的释放时间内停止（参见 §5.4）。

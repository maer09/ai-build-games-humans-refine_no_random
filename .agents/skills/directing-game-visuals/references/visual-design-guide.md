# 视觉设计指南

一份以视觉标签为创意种子，指导动作类小游戏视觉层设计的指南。与 `designing-mini-games` 技能（涵盖机制设计）并行，本文档覆盖画面构图、渲染风格、反馈效果和视觉一致性。

## 1. 设计挑战

- 以最少的美术素材建立视觉上独特的身份。
- 确保游戏可读性，每个物体的角色必须一目了然。
- 仅通过视觉让动作反馈（命中、得分、危险）令人满足。
- 在融合多个标签方向时保持视觉一致性。

## 2. 四项核心视觉原则与评估标准

### (1) 可读性

- 原则：玩家必须能瞬间区分自己的角色、威胁、可收集物和安全区域。使用颜色、亮度、大小和运动的对比，而非标签或图标。
- 评估：首次接触的玩家能否在看到物体的 2 秒内识别其角色？

### (2) 反馈清晰度

- 原则：每个有意义的游戏事件（得分、受伤、擦身而过、状态变化）必须产生可见的响应。视觉响应的强度应与事件的强度匹配。
- 评估：玩家能否不看计分板就判断发生了什么？"得 1 分"和"得 10 分"在视觉上是否有明显区别？

### (3) 美学一致性

- 原则：所有视觉元素（物体、背景、粒子、UI）必须感觉属于同一个世界。共享的渲染风格（描边粗细、色温、运动质感）统一画面。
- 评估：是否有任何元素在视觉上显得格格不入？移除任何一个元素是否会破坏视觉身份？

### (4) 动态生命力

- 原则：屏幕上不应有完全静止的东西。微妙的运动（呼吸、脉动、飘动）让世界保持活力。物体的运动应传达物理感和重量感，而不仅仅是位置变化。
- 评估：当玩家没有操作时，游戏是否仍然感觉充满活力？物体是否具有质量和惯性？

## 3. 视觉标签的作用

视觉标签遵循与 `designing-mini-games` 技能中机制标签相同的哲学：

- **刺激而非约束**：标签暗示视觉方向；最终设计可以偏离标签。
- **矛盾即机遇**：冲突的标签能创造独特的视觉身份（见 §7）。
- **允许偏离**：最终视觉无法仅用标签来解释也没关系。
- **目的**：防止 LLM 默认使用通用视觉风格。

## 4. 视觉标签分类

### 4.1 分类概览

| 类别 | 角色 | 设计影响 |
|:---|:---|:---|
| `render-*` | 物体的绘制方式 | 所有游戏对象的描边风格、填充、轮廓处理 |
| `geometry-*` | 形状语言 | 定义视觉词汇的基元和构图规则 |
| `motionviz-*` | 运动可视化 | 运动、冲击和能量的可视化方式 |
| `background-*` | 背景处理 | 动态背景风格和大气深度 |
| `lighting-*` | 光照与氛围 | 边缘辉光、深度线索、雾效、泛光合成 |
| `analog-*` | 有机瑕疵 | 抖动、闪烁、噪点、色差伪影 |
| `typography-*` | 文字集成 | 分数、标签和字形在游戏世界中的存在方式 |
| `composition-*` | 画面布局 | 空间组织、留白、视觉层级 |

> 注意：排版实现和字体授权策略属于目标项目对应的引擎/平台专用排版技能范畴。

### 4.2 分类交互图

各分类并非独立存在。典型协同效应：

| 组合 | 效果 |
|:---|:---|
| `render-*` + `lighting-*` | 渲染风格定义边缘；光照在此基础上添加氛围 |
| `geometry-*` + `composition-*` | 形状词汇填充构图定义的空间结构 |
| `motionviz-*` + `analog-*` | 运动效果从有机瑕疵中获得质感 |
| `background-*` + `lighting-*` | 背景和光照共同定义深度/氛围层 |
| `typography-*` + `geometry-*` | 当两个标签同时存在时，字形成为几何实体 |

## 5. 从标签到视觉设计的工作流程

在核心机制确定之后，按以下顺序进行设计。

1. **标签解读**：阅读每个标签的 `description` 和 `keywords`。用语言描述它们所暗示的情绪、质感和运动特征。
2. **跨标签综合**：找到统一所有已选标签的视觉主题。用一句话表达（例如"脉动的霓虹生物"、"精密的几何钟表"）。
3. **机制整合**：识别视觉风格与游戏机制的交汇点：
   - 哪种视觉技术最能传达核心机制？
   - 哪些游戏事件值得最强的视觉响应？
   - 视觉风格是否暗示了机制设计中未考虑的反馈机会？
4. **调色板决策**：从标签情绪中推导 3-5 种颜色。为每种颜色分配游戏玩法角色。典型角色包括玩家、威胁、背景、正面反馈、警告，但需根据游戏实际需求调整（例如，有状态切换的游戏可能需要状态A/状态B，而非固定的威胁/警告）。
5. **图层结构**：定义视觉深度层级：
   - 背景（氛围层，低对比度）
   - 游戏区域（中间层，主要动作）
   - 前景效果（粒子、闪光、UI）
6. **反馈映射**：为每个游戏事件，使用标签风格定义视觉响应：
   - 得分增加 → （例如，叠加辉光爆发、涟漪扩散）
   - 受伤/游戏结束 → （例如，色差分裂、屏幕震动）
   - 擦身而过 → （例如，微妙的边缘闪光、轨迹增强）
   - 状态变化 → （例如，调色板切换、几何变换）
7. **检查清单验证**：使用 §10 的检查清单进行确认。

## 6. 机制 × 视觉整合模式

视觉设计不是装饰，它传达游戏状态。以下模式展示视觉标签如何直接服务于游戏玩法。

### 6.1 游戏手感技巧

将以下技巧应用于**所有物体**（玩家、敌人、障碍物、道具），而不仅仅是玩家。在所有物体上保持一致的应用能创造一个协调的"生动"世界。

| 技巧 | 描述 | 相关视觉标签 |
|:---|:---|:---|
| **挤压与拉伸** | 在碰撞、跳跃、着陆时使物体变形。静止的物体微妙地呼吸。 | `motionviz-elastic-deformation`, `analog-micro-jitter` |
| **动态倾斜** | 物体朝运动方向倾斜。旋转的敌人随速度旋转。 | `motionviz-rotation-reactive`, `geometry-diagonal-dominance` |
| **残影轨迹** | 快速运动时留下渐隐的副本。同样应用于敌人和弹射物。 | `motionviz-afterimage-trail`, `analog-frame-blend` |
| **冲击粒子** | 在碰撞、摧毁、着陆、生成时散射碎片。 | `motionviz-impact-ripple`, `background-particle-layer` |
| **辉光累积** | 逐渐增强辉光以预示危险或充能状态。 | `motionviz-energy-glow-build`, `lighting-additive-glow` |

### 6.2 机制到视觉的映射示例

| 机制标签 | 视觉标签 | 组合效果 |
|:---|:---|:---|
| `player-rotate` | `motionviz-velocity-hue-shift` | 颜色随旋转速度变化，越快色温越暖 |
| `weapon-ray` | `render-glow-outline` | 激光渲染为明亮发光线条，带有泛光光晕 |
| `obstacle-chase` | `motionviz-afterimage-trail` | 追踪敌人留下显示轨迹的幽灵残影 |
| `on_holding-charge` | `motionviz-energy-glow-build` | 按住按钮时玩家周围逐渐累积可见辉光 |
| `field-auto_scroll` | `background-flow-lines` | 滚动方向通过流动的流线可视化 |
| `rule-combo_multiplier` | `typography-numeric-focus` | 连击计数器随倍率增大尺寸/亮度 |
| `on_pressed-reverse_state` | `analog-chromatic-offset` | 状态翻转触发全屏短暂的 RGB 分裂 |
| `player-bounce` | `motionviz-elastic-deformation` | 弹跳物体在接触时挤压，在空中拉伸 |

## 7. 视觉标签的矛盾与创意张力

当给定矛盾的视觉标签时，创造一种统一风格，而不是选择其中之一。

| 矛盾 | 常规做法 | 创意解读 |
|:---|:---|:---|
| `render-wireframe-lines` + `lighting-additive-glow` | 选择线框或辉光 | 发光的线框线条，发光的骨骼形态 |
| `composition-negative-space` + `geometry-primitive-modularity` | 稀疏或密集 | 密集集群孤立在广阔的空白空间中，"复杂性的岛屿" |
| `analog-micro-jitter` + `geometry-grid-alignment` | 有机或几何 | 网格锁定的位置伴随着模拟振动颤抖，"不安的秩序" |
| `background-noise-field` + `composition-centered-stage` | 嘈杂或干净 | 干净的中心被嘈杂的边缘包围，噪点作为暗角 |
| `typography-kinetic-motion` + `render-uniform-stroke` | 动态文字或静态线条 | 文字运动但保持统一描边，"有纪律的动态" |

**原则**：与机制标签相同，不要解决矛盾，而是创造一个新概念使两者同时成立。

## 8. 实现交接

本指南定义视觉方向和反馈语义，而非引擎实现。
选择引擎后，将视觉方案转化为该引擎的渲染基元。

引擎特定的实现示例属于引擎特定技能或项目参考，不属于本视觉方向指南。
